# Delivery Plan

Last updated: 2026-04-11
Status: active

## Control rule

The project runs as a delivery program, not as an open-ended chain of local slices.

At any moment there must be:

- one active stage;
- one active delivery milestone;
- one active Codex packet.

## Stage A - Usable end-to-end engine

Goal: prove one usable path from analysis to an export-ready content package.

Entry:
- project creation through rough-cut preparation already exists;
- the repository has completed the governance reset;
- the runtime is still structurally concentrated in a few files.

Exit:
- the runtime is structurally ready for growth;
- one real builder exists;
- one real export package exists;
- one end-to-end acceptance scenario proves the path.

Stop condition:
- a user can run one real project from analysis text to a usable content artifact.

## Stage B - Builder outputs

Goal: expand the number and breadth of supported output builders.

Entry:
- Stage A output path is real and acceptance-proven.

Exit:
- the product supports multiple practical builders, targeting at least long video, shorts/reels, carousel, and packaging-ready output families.

Stop condition:
- output coverage is broad enough that the product no longer depends on a single proof builder for its value.

## Stage C - Production hardening

Goal: make the software durable enough for repeatable production use.

Entry:
- Stage A path is proven;
- Stage B output breadth exists.

Exit:
- recovery and reload behavior are trustworthy;
- UX and validation are stronger;
- export/package quality is hardened;
- release preparation is credible.

Stop condition:
- the MVP is operationally ready against the release criteria.
